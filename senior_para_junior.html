Mês 6–12 — Projetos mais complexos

O que o senior vai cobrar — sem filtro

  • Código reproduzível. Se você rodar o script duas vezes e o resultado mudar, você tem problema. Use seeds fixas, salve versões dos dados.
  • Documentar o porquê, não o quê. O código já mostra o quê. A documentação explica por que você escolheu aquela metodologia.
  • Saber os resultados de cor. Quando o parceiro perguntar "qual o KS do modelo?", você não pode abrir o notebook ao vivo.
  • Não inventar respostas. "Não sei, mas vou verificar" é melhor do que uma resposta errada dita com confiança.
  • Declarar limitações. Todo modelo tem fraquezas. Quem não declara as suas perde credibilidade quando elas aparecem em produção.

Por que logística e não Random Forest (pergunta garantida na entrevista)

Regulatório (Basel III / IFRS 9): o banco precisa explicar para o regulador por que o modelo tomou uma decisão. Random Forest não tem essa explicabilidade.

Contestabilidade: se um cliente for negado e contestar, o banco precisa mostrar quais variáveis foram usadas e com qual peso.

Scorecard: a logística é convertida em pontos (300–850) que o analista de crédito entende. Floresta de 500 árvores não é convertida em nada humano.

Quando usar árvores: internamente, para feature selection antes da logística, ou onde interpretabilidade não é exigida por lei.

Módulo 0 — Pipeline de Data Science

Antes de qualquer modelo, você precisa entender o processo completo. Todo projeto na MS segue estas 7 etapas — sem exceção.

Por que este módulo existe: você pode saber fazer logística, Monte Carlo e ARIMA. Mas se você receber um problema vago de negócio e não souber transformá-lo em projeto de DS, você trava no dia 1. Este módulo ensina o processo que rodeia o código.
1
Definição do Problema — Transformar negócio em DS
Etapa mais importante. Onde 80% dos projetos fracassam antes de começar.

O que acontece: o gerente chega e diz "quero aumentar o lucro do cartão de crédito". Você NÃO pode simplesmente abrir o Python e rodar um XGBoost "para ver no que dá". Você precisa fazer perguntas até saber EXATAMENTE qual é o problema real.

Por que: "aumentar o lucro do cartão" pode ser 4 projetos completamente diferentes:

"Estamos perdendo muito com fraudes" → Problema DS: Detecção de Anomalia / Classificação em tempo real → Modelo: Random Forest ou XGBoost (velocidade importa) → Métrica: Recall alto (não deixar fraude passar) "Clientes estão cancelando o cartão" → Problema DS: Previsão de Churn (quem vai cancelar nos próximos 30 dias) → Modelo: Logística (precisa de explicabilidade para marketing) → Métrica: Recall alto (FN = perder cliente = caro) "Não estamos conseguindo vender o cartão" → Problema DS: Modelo de Propensão (quem tem perfil para aceitar oferta) → Modelo: Logística ou Gradient Boosting → Métrica: Precision (se for brinde caro, não quer FP) "Clientes usam pouco o cartão" → Problema DS: Segmentação (clustering por comportamento de uso) → Modelo: K-Means → Métrica: Silhouette Score + interpretação de negócio

Sua missão na Etapa 1: fazer perguntas até o gerente admitir que o problema real é, por exemplo, o churn. Só depois você vai para a Etapa 2.

3.1 — Dados Faltantes (Missing Values)

SoluçãoQuando usarProblema
Remover a linhaMenos de 5% de missingPerde dados. Se o missing tiver padrão (autonomos não declaram renda), você cria viés.
Imputar com medianaVariável numérica com <30% missingÉ uma "mentira" — você inventa valores. Reduz variabilidade.
Modelo de imputaçãoMissing >15% e variável importanteMais preciso. Usa outras colunas para prever o valor faltante. Complexo.
Por que mediana e não média? A média é puxada por outliers. Um CEO que ganha R$500k eleva a média de renda de toda a base. A mediana não se move com outliers — é o valor do meio.

3.2 — Outliers

Problema: cliente com "idade = 210" ou "salário = R$ 500 milhões" O modelo vai tentar se ajustar a esses pontos impossíveis e fica distorcido. Solução 1 — Remover: se for claramente erro de digitação Solução 2 — Clipar (Winsorize): define um limite razoável Ex: qualquer renda acima do percentil 99 → trata como se fosse o P99 df['renda'] = df['renda'].clip(upper=df['renda'].quantile(0.99)) Solução 3 — Transformação Log: para variáveis muito assimétricas (renda, valor de transação) df['log_renda'] = np.log1p(df['renda']) → "achata" a escala, deixa os outliers menos impactantes

3.4 — Feature Scaling (Colocar na mesma régua)

Problema: "Idade" vai de 20 a 80. "Salário" vai de 2.000 a 200.000. Modelos como regressão logística vão achar que Salário é mais importante só porque o número é maior.

StandardScaler (Padronização — Z-Score): MAIS COMUM z = (valor - média) / desvio_padrão → Resultado: média=0, desvio=1 → Robusto. Use em logística, redes neurais, k-means. MinMaxScaler (Normalização): z = (valor - mínimo) / (máximo - mínimo) → Resultado: sempre entre 0 e 1 → Sensível a outliers. Use se você precisa de 0-1 explicitamente. IMPORTANTE: Árvores (Random Forest, XGBoost) NÃO precisam de scaling. Elas não usam distâncias, usam cortes. Só escale para logística, KNN e redes.

Módulo 1 — Credit Scoring

O modelo mais comum que você vai construir. Aprenda este do jeito certo.

Este módulo é 50% do seu trabalho no primeiro ano. Modelos de PD (Probabilidade de Default) para carteiras de crédito pessoal, empresarial e imobiliário.

Pipeline completo de um scorecard

1

EDA — Conhecer os dados antes de qualquer coisa

Pé no chão: Você não opera sem conhecer o paciente. O mesmo vale para dados.

O que olhar: distribuição de cada variável (tem muitos zeros? Tem outliers de R$999.999?), quanto dado está faltando (missing >40%? Descarta a variável), e principalmente: qual a taxa de default?

Por que a taxa importa: Se apenas 3% dos clientes calotam, o modelo vai aprender que é "mais seguro" sempre dizer "não vai calotar" — e vai acertar 97% das vezes sem ser útil. Esse é o problema de desbalanceamento de classes. Você precisa saber disso antes de treinar.

2

WoE — Weight of Evidence (Peso de Evidência)

Nível 1 — A ideia em uma frase: Para cada faixa de uma variável, o WoE responde: "Essa faixa concentra mais caloteiros ou mais bons pagadores do que a média do portfólio?"

Nível 2 — Analogia do bairro: O banco quer saber se morar em um bairro específico é sinal de risco. O WoE calcula: a proporção de caloteiros nesse bairro é maior ou menor do que a proporção média na cidade inteira?

  • Bairro com muito mais caloteiros que a média → WoE negativo (perigo)
  • Bairro com muito menos caloteiros que a média → WoE positivo (seguro)
  • Bairro igual à média → WoE = 0 (indiferente)

Nível 3 — A Fórmula:

WoE
WoEi = ln( %_bons_na_faixa_i ÷ %_maus_na_faixa_i )

Onde %_bons_na_faixa_i = (bons nessa faixa) ÷ (total de bons do dataset) — não é a % dentro da faixa, é a % dentro do grupo de bons.

Por que logaritmo? O ln torna a escala simétrica: WoE de +1 (dobro de bons) e −1 (dobro de maus) são igualmente distantes de zero. Além disso, o WoE tem relação linear com o logit da regressão logística — é exatamente o que ela precisa.

Por que não usar a variável crua? A logística assume que o efeito de +R$1 na renda é constante em todo o intervalo. Isso é falso: a diferença de R$500 → R$1.000 é enorme para o risco; R$30.000 → R$30.500 é irrelevante. O WoE captura essa não-linearidade agrupando clientes com comportamento de risco similar.

Bônus: O WoE trata missing automaticamente — crie um bin "ausente" e calcule o WoE desse bin com os dados reais. Sem imputação artificial.
3

IV — Information Value (Valor de Informação)

Se o WoE mede o risco de cada faixa, o IV mede o poder preditivo total da variável inteira. É a soma dos IV parciais de cada faixa.

IV
IV = Σ (%_bons_i − %_maus_i) × WoE_i  (soma sobre todas as faixas)

Intuição: Uma faixa contribui muito para o IV quando tem duas características juntas: (1) grande diferença entre %bons e %maus, e (2) WoE alto em módulo. Se a faixa tem exatamente a proporção média, %bons = %maus → diferença = 0 → IV parcial = 0.

IV < 0.02 → ❌ Descarta. Não sabe distinguir bons de maus. IV 0.02–0.10 → ⚠️ Fraco. Entra só se não houver opção melhor. IV 0.10–0.30 → ✅ Boa. Use sempre. IV 0.30–0.50 → ✅ Muito forte. Excelente variável. IV > 0.50 → 🔴 Investigue leakage! Quase nunca acontece de forma legítima.

Leakage (vazamento): IV acima de 0.5 indica que a variável pode conter informação do futuro. Exemplo real: "flag de renegociação" tem IV 0.65. Parece ótimo — mas essa flag só existe depois que o cliente entrou em atraso. Usar no modelo é como olhar a resposta antes da prova.

Células com zero: Se uma faixa tem n_maus = 0, então WoE = ln(X/0) = infinito. Solução: adicione 0.5 a todos os zeros antes de calcular (suavização de Laplace), ou agrupe a faixa com a vizinha.

4

Regressão Logística — Treinar o modelo

Tradução: A logística é uma balança. De um lado os sinais de risco (coeficiente positivo = mais chance de calote). Do outro, sinais de bom pagador (coeficiente negativo). A balança pesa tudo e entrega uma probabilidade de 0 a 1.

Por que logística e não Random Forest? Em ambiente regulado (bancos, seguradoras), a lei exige que você explique ao cliente negado e ao regulador: quais variáveis pesaram e quanto. Uma floresta com 500 árvores não tem essa explicação. A logística tem coeficientes claros: "variável renda teve coeficiente -0.8 — quem tem renda alta tem menos risco."

Use statsmodels, não sklearn, quando precisar de p-valores para o relatório regulatório. O sklearn é rápido mas não gera a tabela de significância estatística que o auditor vai pedir.

Coeficiente positivo → aumenta a probabilidade de default Coeficiente negativo → diminui a probabilidade de default Score = -ln(PD / (1 - PD)) × fator + offset → Típico: escala de 300 a 850 pontos (quanto maior, melhor o cliente)
5

Validação — KS, Gini, AUC

Tradução: O modelo sabe separar quem vai pagar de quem não vai?

Você valida sempre no teste (dados que o modelo nunca viu). Se o KS do treino for 50% e do teste for 25%, o modelo decorou os dados de treino — não aprendeu a regra. Isso é overfitting.

Diferença aceitável entre treino e teste: até 5 pontos de KS. Acima disso, regularize (C menor no sklearn, ridge na logística).

KS treino = 45%, KS teste = 42% → ✅ Modelo generaliza bem KS treino = 55%, KS teste = 30% → ❌ Overfitting — decorou o treino
6

PSI — Monitorar o modelo em produção

Tradução: O mundo mudou depois que eu treinei o modelo?

Analogia: Seu modelo foi treinado com clientes de 2022. É 2024. Houve uma crise, a Selic subiu de 2% para 13%, o perfil dos clientes mudou. Seu modelo ainda é válido? O PSI responde essa pergunta comparando a distribuição dos scores de hoje com a distribuição de quando o modelo foi criado.

PSI < 0.10 → ✅ Estável. Modelo continua representando a realidade. PSI 0.10–0.20 → ⚠️ Alerta. Investigar o que mudou no perfil. PSI > 0.20 → 🔴 Retreinar. O modelo não representa mais os clientes atuais. Causas comuns: mudança de canal de aquisição, crise econômica, nova legislação, inflação corroendo o poder de compra.

Verificando a Monotonicidade

Após calcular os WoEs, plote-os. Devem seguir uma sequência sempre crescente ou sempre decrescente. No nosso exemplo:

✅ WoE monotônico crescente (correto — faz sentido econômico): Até R$1k → WoE −0.69 (zona mais perigosa) R$1k–3k → WoE −0.36 (menos perigoso) R$3k–6k → WoE +0.52 (zona segura) Acima R$6k → WoE +0.74 (mais seguro) ❌ WoE não-monotônico (exemplo de problema): Até R$1k → WoE −0.69 R$1k–3k → WoE +0.20 ← subiu? Não faz sentido. R$3k–6k → WoE −0.10 ← caiu? Economicamente absurdo. Acima R$6k → WoE +0.74 Solução: agrupe R$1k–3k com R$3k–6k num bin único e recalcule.

Monotonicidade garante que o modelo aprende a regra econômica real — não memoriza ruído estatístico dos dados de treino.

Módulo 3 — Monte Carlo

  
Quando você não consegue calcular uma distribuição analiticamente, você a simula.


  
Por que simular? Por que não calcular direto?

  
Analogia: Você quer saber a probabilidade de chover mais de 100mm durante um festival de 3 dias em São Paulo. A matemática analítica consegue calcular, mas é complicada. A alternativa: simule 100.000 festivais, conte em quantos choveu mais de 100mm. Essa porcentagem é a probabilidade que você queria.

  
Em finanças: um portfólio com 10 ações tem uma distribuição de perdas que depende de como essas ações se correlacionam. Calcular isso analiticamente é trabalhoso. Simular é direto: gere 100.000 cenários de retornos, calcule o P&L de cada um, olhe o percentil 5% — esse é o VaR a 95%.


  
A ideia central: jogue o dado 100.000 vezes. O percentil 5% das simulações é o VaR a 95%. Quanto mais simulações, mais preciso o resultado.


  
VaR vs CVaR — a diferença que o regulador exige

  
VaR 95%  → "Com 95% de certeza, não perco mais que R$X amanhã."
              Mas o que acontece nos 5% ruins? O VaR não diz.

CVaR 95% → "Nos 5% piores dias, perco em média R$Y."
              Y > X sempre. CVaR é mais conservador.

Por que Basel III exige CVaR:
  → O VaR diz onde começa o problema, não o tamanho dele.
  → Em crises (2008, 2020), as perdas são 3-5x o VaR histórico.
  → CVaR captura o tamanho da catástrofe, não só o limiar.


  
Por que Cholesky? (pergunta técnica da entrevista)

  
Problema sem Cholesky: Se você gera os retornos de PETR4 e VALE3 de forma independente, está dizendo que elas não se influenciam. Mas na realidade, quando o petróleo cai, PETR4 cai, e o mercado em geral cai, então VALE3 também tende a cair. Ignorar isso subestima o risco de carteira.

  
Analogia: Simular o clima de São Paulo: temperatura e umidade são correlacionadas (dia quente tende a ser seco). Se você simula as duas de forma independente, vai criar "dias" com temperatura de 40°C e umidade de 95% ao mesmo tempo — que é fisicamente impossível. Cholesky injeta a correlação histórica para que as simulações sejam fisicamente coerentes.

  
Sem Cholesky:  Z = ruído aleatório independente
                 Cada ativo sorteado sem olhar os outros
                 → Subestima o risco porque ignora que os ativos caem juntos

Com Cholesky:  L = Cholesky(matriz_covariância)
               R = μ + Z × L.T
               → Os retornos simulados têm a mesma correlação histórica
               → Se PETR4 cai muito, VALE3 também tende a cair na simulação

L × L.T = Σ (a decomposição "distribui" a correlação nos dados gerados)

Limitação do Monte Carlo: as simulações usam a correlação histórica. Em crises sistêmicas, a correlação sobe para 1 (tudo cai junto) e o Monte Carlo subestima as perdas. Por isso sempre roda junto um stress test com cenários históricos de crise (2008, 2020, 2002).







  
Módulo 4 — GLM para Risco

  
A regressão logística é um GLM. Mas existem outros para LGD e frequência de sinistros.


  
Por que não usar regressão linear para tudo?

  
Problema 1 — Valores impossíveis: Regressão linear pode prever LGD = -0.3 (perda negativa — o banco ganha com o calote?) ou LGD = 1.4 (perda de 140%?). Matematicamente válido. Financeiramente absurdo.

  
Problema 2 — Distribuição errada: A regressão linear assume que o erro tem distribuição Normal (simétrica, mesma variância em todo lugar). Mas LGD tem distribuição assimétrica — muitos clientes com LGD baixo (recuperação razoável), poucos com LGD altíssimo (perdas totais). Forçar Normal aqui é mentir para o modelo.

  
A solução — GLM: Você escolhe a família de distribuição que melhor representa o seu target. O modelo se adapta à realidade dos dados, não o contrário.


  
GLM = Generalized Linear Model. Você escolhe a família de distribuição do target e a função de ligação (link) que conecta os preditores à média esperada. A regressão logística que você já conhece é um GLM com família Binomial e link Logit.


  
Quando usar cada família — com analogia

  


    
B

      
Binomial + Logit → target 0 ou 1

      
Quando usar: O target é uma decisão binária. Calotou ou não. Fraude ou não. Churned ou não.

      
Analogia: Jogar moeda. O resultado só pode ser cara ou coroa. A logística estima a probabilidade de cara dado o perfil do cliente.

      
Saída: Probabilidade entre 0 e 1 (nunca fora disso). É o score de PD.

    


    
P

      
Poisson + Log → contagem de eventos

      
Quando usar: O target conta coisas inteiras que só podem ser positivas: número de atrasos, número de sinistros no ano, número de ligações de cobrança.

      
Analogia: Contar carros que passam por um pedágio em uma hora. Pode ser 0, 1, 50, 200. Mas nunca -3 ou 2.7. Distribuição Poisson representa isso perfeitamente.

      
Interpretação: exp(coeficiente) = fator multiplicador sobre a contagem esperada.

    


    
G

      
Gamma + Log → valor positivo e assimétrico

      
Quando usar: O target é um valor contínuo que só pode ser positivo e tem distribuição com cauda longa à direita. LGD (0 a 1, com pico próximo de 0.6), severidade de sinistro (maioria pequena, poucos gigantes), valor de indenização.

      
Analogia: Tempo de espera em fila de banco. Nunca negativo. A maioria espera 5-10 minutos. Mas algumas pessoas esperam 45 minutos por azar. Distribuição Gamma representa isso — assimétrica com cauda longa.

      
Interpretação: exp(coeficiente) = fator multiplicador sobre o LGD esperado. Coef = -0.8 → exp(-0.8) = 0.45 → ter garantia reduz o LGD em 55%.

    


    
T

      
Tweedie → zeros + valor contínuo

      
Quando usar: O target tem muitos zeros misturados com valores positivos contínuos. Custo total de sinistros em seguros: a maioria das apólices não tem sinistro (custo = 0), mas algumas têm sinistros caros.

      
Analogia: Conta de médico no mês: a maioria dos meses é zero (você não foi ao médico). Mas o mês que você foi, pode ter custado R$500 a R$3.000. Tweedie modela os zeros e os valores positivos ao mesmo tempo.

    


  


  

    
Python — Modelo de LGD com GLM Gamma

    

  

  
LGD com zeros e uns exatos: use modelo em dois estágios. Primeiro logística classifica LGD=0 vs LGD>0 (recuperação total vs alguma perda). Depois Gamma modela os valores entre 0 e 1. O resultado final combina os dois: P(perda) × LGD_esperado.







  
Módulo 5 — Séries Temporais

  
Forecast de inadimplência, previsão de provisões, trend models da vaga.


  
O que é uma série temporal?

  
Tradução: Qualquer medida que você observa repetidamente ao longo do tempo na mesma ordem. A inadimplência do banco em janeiro, fevereiro, março... dezembro. A ordem importa. Você não pode embaralhar os meses como faria com um dataset de clientes.

  
Por que é diferente de modelagem tradicional? Em um scorecard, você pode pegar os clientes, embaralhá-los, separar treino e teste aleatoriamente. Em série temporal, o futuro não pode estar no treino — isso seria data leakage temporal. Se você treina com 2020-2023 e testa com 2022, o modelo "viu" o futuro. O resultado parece ótimo mas é uma mentira.


  
Pipeline correto — sempre nessa ordem

  


    
1

      
Decomposição — Desmontar a série em peças

      
Tradução: Toda série temporal é a soma de 3 coisas: Tendência + Sazonalidade + Ruído. Você separa as peças para entender o que está acontecendo.

      
Tendência: A direção geral. Vista de longe, a inadimplência está subindo? Caindo? Estável? É o "drift" do negócio.

      
Sazonalidade: O padrão que volta todo ano na mesma época. Inadimplência sempre sobe em janeiro/fevereiro (pós-natal + pós-carnaval). Se você não souber disso, vai achar que o banco vai à falência todo início de ano.

      
Resíduo (Ruído): O que sobra depois de remover tendência e sazonalidade. Regra de ouro: se o resíduo tiver padrão, o modelo está errado — ainda há informação que você não extraiu. Se parecer estática aleatória, você extraiu tudo.

    


    
2

      
Teste ADF — A série está "quieta"?

      
Tradução: Estacionária = série que tem média e variância estáveis ao longo do tempo. O ARIMA precisa que a série seja estacionária para funcionar direito.

      
Analogia: Uma série estacionária é como o batimento cardíaco de alguém dormindo: sempre entre 55 e 65 bpm, vai e volta, sem tendência. Uma série não-estacionária é o batimento de alguém correndo uma maratona: a média sobe conforme o esforço aumenta.

      
O que fazer se não for estacionária: serie.diff() — você subtrai o valor de hoje pelo de ontem. Isso remove a tendência. Repita até o ADF dar p-valor < 0.05. Esse número de diferenciações é o "d" do ARIMA(p, d, q).

      
p-valor ADF < 0.05 → ✅ Estacionária. Pode usar ARIMA.
p-valor ADF > 0.05 → ❌ Não estacionária. Diferencie: serie.diff()

    


    
3

      
ARIMA(p, d, q) — O modelo de previsão

      
Tradução simples: O ARIMA usa o passado para prever o futuro, considerando tanto os valores anteriores (AR) quanto os erros de previsão anteriores (MA).

      
p → AR (AutoRegressivo): quantos períodos passados usamos como input?
     Se a inadimplência de 3 meses atrás ainda influencia hoje: p=3

d → Diferenciação: quantas vezes diferenciamos para estabilizar?
     Resultado do teste ADF.

q → MA (Média Móvel dos Erros): quantos erros passados corrijo?
     Ajuda o modelo a "aprender com os próprios erros" de previsão.

Na prática: use auto_arima(serie, seasonal=True, m=12)
Ele testa as combinações e escolhe pelo AIC (menor AIC = melhor).
Você não precisa adivinhar p, d, q manualmente.

    


    
4

      
Validação dos Resíduos — O modelo capturou tudo?

      
Tradução: Plote os erros (resíduos) do modelo. Se parecerem estática aleatória — parabéns. O modelo extraiu toda a informação da série. Se houver padrão (ondas, tendência, agrupamentos) — o modelo está falhando, ainda tem informação que ele não capturou.

      
Teste de Ljung-Box: testa se os resíduos são ruído branco. p-valor > 0.05 = resíduos são independentes = modelo capturou tudo.

      
Intervalos de confiança: o ARIMA gera previsão + intervalo de confiança. O intervalo se abre como um cone conforme avança no tempo — a incerteza cresce com o horizonte de previsão. Previsão de 1 mês: intervalo estreito. Previsão de 12 meses: intervalo largo. Isso é esperado e correto.

    


  


  

    
Python — Pipeline completo

    

  

  
Nunca embaralhe dados de série temporal. O futuro não pode estar no treino. Use sempre holdout temporal: treino = todo o período até data X, teste = período posterior a X. Nunca train_test_split com shuffle=True em série temporal.
